# 目标

构建一个完整的桌面端角色扮演聊天应用，满足以下核心能力：

- 支持单角色聊天，并以 `SSE` 方式向前端持续推送流式回复
- 支持多角色群聊，支持轮流发言与 `@角色` 定向发言
- 支持长期记忆、短期记忆和摘要记忆，其中长期记忆必须全量 ES 化
- 支持 Elasticsearch 混合检索，融合 `BM25 + dense_vector + 标签过滤/增强`
- 支持角色背景、原作语料、段落上下文、标签信息共同参与回答增强
- 支持工程化开发、测试、打包和文档交付

技术栈固定且不得替换：

- `Electron`
- `Vite`
- `React`
- `TypeScript`
- `LangGraph`
- `SQLite`
- `DeepSeek`
- `Elasticsearch`

## 实施方案映射

本文件以 `Agent实施方案.md` 为能力蓝本，但落地到当前固定技术栈时必须遵循以下映射：

- 实施方案中的 Python Agent 编排，改为 `TypeScript + LangGraph` 实现
- 实施方案中的本地服务流式输出，改为桌面端内的 `SSE` 流式通道
- 实施方案中的 SQLite 记忆层，升级为 `SQLite 持久化 + ES 长期记忆检索索引`
- 实施方案中的三路混合检索，改为 Electron 项目中的 ES 检索服务模块实现
- 实施方案中的多角色能力，落地为桌面端群聊和多角色协同回复能力

## 角色策略

默认以《千恋万花》的角色扮演场景为目标，角色策略如下：

- 可扮演核心角色默认为 `丛雨`、`芳乃`、`茉子`、`蕾娜`
- `将臣` 默认作为用户视角、关系对象和上下文角色保留
- 系统必须支持扩展角色，但首版最低要求先确保以上角色工作正常
- 角色约束、称呼规则、关系信息、语言风格必须可配置，不得硬编码在单个 UI 组件中

## Agent 角色

执行本文件的 agent 必须完成以下职责：

- 按阶段推进实现，而不是一次性无结构生成全部代码
- 在关键输入缺失时先询问用户，不得自行虚构密钥、账号、地址或索引数据
- 优先交付最小可运行闭环，再补足检索增强、记忆增强、测试和文档
- 每一阶段结束后，代码必须仍然可运行、可构建或可验证
- 保证所有敏感配置均来源于环境变量，不得硬编码
- 保证长期记忆检索、SSE 流式输出、标签增强检索、多角色群聊被明确实现，而不是仅写在说明中

## 核心架构

### 1. 检索架构

检索层必须采用混合检索，不得只做单一向量检索或单一关键词检索。

最低要求：

- 路 1：`dense_vector` 语义检索
- 路 2：`BM25` 文本检索
- 路 3：标签增强检索或标签过滤增强
- 融合方式至少支持 `RRF` 或等价排序融合策略

标签增强检索要求：

- 必须使用 `dialogue_tags.jsonl` 与 `passage_tags.jsonl`
- 至少支持 `scene`、`emotion`、`function` 三类标签过滤或加权
- 必须允许按 `character`、`is_playable`、`record_type`、`chapter` 做结构化过滤
- 检索结果中必须同时支持单句与段落记录

### 2. 记忆架构

记忆层必须采用三层结构：

- 短期记忆：最近若干轮对话，直接进入 prompt
- 摘要记忆：对话阶段性压缩摘要
- 长期记忆：重要事件、偏好、关系变化、承诺事项

长期记忆全量 ES 化要求：

- 长期记忆不能只存 SQLite
- SQLite 作为事实持久化源保留
- Elasticsearch 作为长期记忆检索索引必须同步写入
- 长期记忆检索必须走 ES，而不是简单 SQL `LIKE`
- 长期记忆至少包含 `session_id`、`character`、`content`、`category`、`timestamp`、`dense_vector`、`tags`

### 3. 流式架构

流式输出必须统一采用 `SSE`：

- 后端必须暴露稳定的流式消息协议
- 渲染进程必须能订阅、消费、取消 SSE 流
- SSE 事件至少包含 `token`、`message_done`、`error`
- 群聊场景下必须能区分当前输出属于哪个角色

在 Electron 中的实现要求：

- 可以使用内置本地 HTTP 服务或安全桥接层承载 SSE
- 不得仅用一次性 IPC 返回完整字符串冒充流式输出
- 若存在 IPC 辅助，只能作为 SSE 建链、控制或兜底，不得替代 SSE 主通道

### 4. 群聊架构

多角色群聊是一级能力，不是附加功能。

最低要求：

- 群聊支持 2 到 5 个角色
- 支持轮流发言模式
- 支持 `@角色名` 指定回复模式
- 每个角色回复前都能获得自身检索上下文和共享会话上下文
- 某角色生成失败时允许跳过，但不得中断整个群聊轮次

## LangGraph 编排要求

LangGraph 必须覆盖以下能力：

- 单聊链路
- 多角色群聊链路
- 长期记忆提取链路
- 检索融合链路
- 合规校验与重试链路

建议状态接口：

```ts
export interface Message {
  id?: string;
  role: "user" | "assistant" | "system";
  content: string;
  roleId?: string;
  chatId?: string;
  timestamp: number;
}

export interface RetrievedDoc {
  sourceId: string;
  recordType: "dialogue" | "passage" | "memory";
  character: string;
  text: string;
  score: number;
  tags?: {
    scene?: string[];
    emotion?: string[];
    function?: string[];
    tone?: string[];
  };
}

export interface ChatState {
  chatId: string;
  messages: Message[];
  participants: string[];
  currentRoleId?: string;
  mode: "single" | "group";
  mentionTarget?: string | null;
  retrievedDocs: RetrievedDoc[];
  longTermMemories: RetrievedDoc[];
  summaryMemory?: string;
  streamId?: string;
  next?: string;
}
```

建议节点：

- `retrieve_context`
- `retrieve_long_term_memory`
- `build_prompt`
- `call_llm_stream`
- `validate_response`
- `save_message`
- `extract_memory`
- `index_memory_to_es`

路由要求：

- 单聊：`user -> retrieve_context -> retrieve_long_term_memory -> call_llm_stream -> validate_response -> save_message -> extract_memory -> end`
- 群聊：用户消息后，按参与者顺序循环执行角色回复子图
- `@角色` 模式下，只触发指定角色或优先该角色
- 某角色回复失败时，记录错误并继续剩余角色

## 执行阶段

agent 必须按以下阶段推进。

### 阶段 1：初始化工程

目标：

- 初始化 Electron + Vite + React + TypeScript 工程
- 补齐主进程、预加载、渲染进程、后端服务目录
- 保证最小窗口可启动

最低验收：

- `npm install` 成功
- `npm run dev` 可启动桌面应用
- 主进程、渲染进程、后端服务职责边界清晰

### 阶段 2：实现数据层与角色配置

目标：

- 建立 SQLite 数据库
- 建立角色、会话、消息、记忆表
- 接入 `character_constraints.json`

最低需要表：

- `characters`
- `chats`
- `messages`
- `memory_events`
- `memory_summaries`

最低验收：

- 数据库初始化完成
- 角色与会话可持久化
- `character_constraints.json` 可被加载并参与 prompt 构造

### 阶段 3：实现 ES 对话索引与标签接入

目标：

- 从对话、段落、标签数据构建 ES 对话索引
- 建立单句和段落统一检索能力

实现要求：

- 必须接入 `dialogues_clean.jsonl`
- 必须接入 `dialogue_passages.jsonl`
- 必须回填 `dialogue_tags.jsonl` 与 `passage_tags.jsonl`
- 文档中必须包含 `all_tags`、`tags.scene`、`tags.emotion`、`tags.function`
- 必须支持 `dense_vector`

最低验收：

- ES 索引创建成功
- 对话与段落数据均可查询
- 标签字段可用于 filter 或 boost

### 阶段 4：实现混合检索服务

目标：

- 封装混合检索服务，统一提供给 LangGraph 调用

实现要求：

- 至少实现 `dense`、`bm25`、`tag-aware` 三路召回
- 支持 `character`、`record_type`、`chapter`、`is_playable` 过滤
- 支持 Top-K 融合返回
- 支持长期记忆索引和对话索引的独立检索

最低验收：

- 可独立调用检索服务返回结构化结果
- 标签过滤和角色过滤可生效

### 阶段 5：实现长期记忆全量 ES 化

目标：

- 实现 SQLite + ES 双写的长期记忆体系

实现要求：

- 新增长期记忆 ES 索引
- 新消息触发记忆提取时，重要记忆同时写入 SQLite 和 ES
- 长期记忆检索必须优先调用 ES
- 必须支持按 `session_id`、`character`、`category` 过滤

最低验收：

- 新增长期记忆后可在后续会话被检索回来
- 关闭应用重启后长期记忆仍可恢复

### 阶段 6：实现 DeepSeek + SSE 流式服务

目标：

- 封装 DeepSeek 调用
- 将生成结果通过 SSE 持续推送到前端

实现要求：

- 通过环境变量读取 API Key 和 Base URL
- 前端逐 token 或逐 chunk 接收流式内容
- SSE 必须支持中断、完成、错误事件
- 群聊时必须携带 `roleId` 或等价字段

最低验收：

- 单聊可见连续流式输出
- 群聊每个角色的流式输出可区分

### 阶段 7：实现多角色群聊

目标：

- 完成多角色轮流发言与 `@角色` 模式

实现要求：

- 群聊创建页可选择 2 到 5 名角色
- 发言顺序默认按 `participants` 数组顺序
- 每个角色回复前执行各自检索与记忆召回
- 前端必须清楚展示当前正在发言的角色

最低验收：

- 群聊一轮回复可完整跑通
- 单个角色失败不会中断整个群聊

### 阶段 8：补齐测试与文档

目标：

- 覆盖高风险链路
- 交付完整说明文档

测试至少覆盖：

- SQLite 数据访问
- ES 检索封装
- 标签增强检索逻辑
- 长期记忆双写与召回
- LangGraph 状态流转
- SSE 流式输出
- 多角色群聊顺序

## IPC 与安全要求

- 预加载层只暴露必要能力
- 渲染进程不得直接访问 Node 高危 API
- 如使用本地 HTTP 服务承载 SSE，必须限制监听地址与访问范围
- 敏感配置不得下发到渲染进程明文存储

## 实现要求

agent 在实现时必须遵守以下规则：

- 使用严格 TypeScript，避免滥用 `any`
- 优先清晰模块边界，不得把索引、检索、记忆、SSE、LangGraph 全部堆到一个文件
- 默认先实现最小闭环，再逐步增强
- 对已有仓库增量修改，不得粗暴覆盖无关文件
- 检索、记忆、群聊、流式必须是真实实现，不得只写接口壳子
- 所有密钥、密码、地址、索引名均从环境变量读取或从模板配置读取
- 对外部服务失败必须有日志、错误提示或降级策略

## 禁止项

agent 不得执行以下行为：

- 不得伪造用户未提供的凭据
- 不得将长期记忆检索简化为 SQLite 关键词匹配后宣称“ES 化”
- 不得将完整响应一次性返回并冒充 SSE
- 不得只实现单角色聊天却声称已支持群聊
- 不得把标签文件接入说明文档而不接入实际查询
- 不得把敏感配置写入示例源码或测试快照

## 质量门禁

最终交付必须满足以下条件：

- TypeScript 编译通过，至少满足 `tsc --noEmit`
- 关键测试可运行，至少包含 `npm test`
- 单聊可正常收发消息
- SSE 流式输出真实可见
- 群聊按 `participants` 顺序轮流发言

- Elasticsearch 连接失败或检索失败时，应用能给出明确错误或降级行为
